ℹAbout Str:::lab Studio

Str:::lab Studio is a browser-based SQL Studio that connects to any Flink SQL Gateway. It provides a complete development, monitoring, and administration experience — without the overhead of a full platform deployment. Works with local Docker, Kubernetes, and any self-hosted Flink cluster.

What it is

A zero-dependency single-file HTML application served by nginx. All SQL execution is driven by the Flink SQL Gateway REST API. Results stream live into the browser. Session state is managed client-side in localStorage.

Who it's for

Streaming data engineers and platform teams who want a proper IDE experience for Apache Flink SQL — query editing, live result streaming, real-time job monitoring, checkpoint tracking, and professional reporting — all from a browser.

Open source

Created by Nestor Martourez A. A — open source under MIT. Community contributions welcome at github.com/coded-streams/strlabstudio.

Kube Operator

The Str:::lab Studio Kube Operator manages Studio deployments as first-class Kubernetes CRDs. Install via Helm from strlabstudio-operator.

🔀Full Pipeline Demo

A complete multi-stage streaming pipeline: datagen source → Kafka → enrichment with branching → windowed aggregation → live preview.

SET execution.runtime-mode = 'STREAMING';
SET parallelism.default = '4';
SET state.backend = 'rocksdb';
SET table.exec.mini-batch.enabled = 'true';
SET table.exec.mini-batch.allow-latency = '2000';
SET table.state.ttl.type = 'IDLE';
SET table.state.ttl.time = '3600000';
SET pipeline.name = 'trading-pipeline-demo';
CREATE TEMPORARY TABLE raw_trades (
  trade_id STRING, symbol STRING, amount DECIMAL(18,2),
  ts AS CURRENT_TIMESTAMP,
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector'='datagen','rows-per-second'='100',
  'fields.symbol.length'='4',
  'fields.amount.min'='100','fields.amount.max'='999999'
);
INSERT INTO kafka_trades SELECT * FROM raw_trades;
INSERT INTO enriched_trades
SELECT trade_id, symbol, amount,
  (amount/100000.0) AS risk_score,
  CASE WHEN (amount/100000.0) >= 0.80 THEN 'CRITICAL'
       WHEN (amount/100000.0) >= 0.55 THEN 'HIGH'
       WHEN (amount/100000.0) >= 0.30 THEN 'MEDIUM'
       ELSE 'LOW' END AS risk_tier, ts
FROM kafka_trades;
INSERT INTO trade_summary
SELECT TUMBLE_START(ts, INTERVAL '10' SECOND), TUMBLE_END(ts, INTERVAL '10' SECOND),
  symbol, COUNT(*), SUM(amount),
  SUM(CASE WHEN risk_tier='CRITICAL' THEN 1 ELSE 0 END),
  SUM(CASE WHEN risk_tier='HIGH' THEN 1 ELSE 0 END)
FROM enriched_trades
GROUP BY TUMBLE(ts, INTERVAL '10' SECOND), symbol;
SELECT trade_id, symbol, amount, risk_tier FROM enriched_trades
WHERE risk_tier IN ('CRITICAL','HIGH') ORDER BY ts DESC;

⨍UDF Manager v1.2.0 — March 2026

The UDF Manager provides the complete User-Defined Function lifecycle — browsing, building, uploading, and registering — without leaving Str:::lab Studio. Access it from the topbar via the ⨍ UDFs button.

Version 1.2.0 (March 2026) resolves the confirmed ClassNotFoundException and "Cannot run program python: error=2" errors through a guided 3-step registration wizard with mandatory classpath verification before CREATE FUNCTION is attempted.

🔴
Two separate JVMs — this is the most important thing to understand The JobManager REST API (port 8081) and the SQL Gateway (port 8083) are separate JVM processes, often in separate Docker containers. Uploading a JAR via Upload JAR copies it to the JobManager container. The SQL Gateway cannot see that file unless both containers share the same filesystem (shared volume mount).

ADD JAR '/path/to/jar' must run inside a SQL Gateway session using a path that exists on the Gateway container. Until this completes successfully, CREATE FUNCTION LANGUAGE JAVA will always throw ClassNotFoundException.

The two UDF paths — which one do you need?

📦 Path A — JAR UDF (Java/Scala/Python)

You have a compiled class extending ScalarFunction, TableFunction, or AggregateFunction. Requires building and uploading a JAR.

Root cause of ClassNotFoundException (confirmed fix in v1.2.0)

⚠
Why you see "Cannot run program python: error=2" This Python error is a red herring. It appears when you register a function with LANGUAGE SCALA or LANGUAGE PYTHON as a workaround — those languages skip Java validation at CREATE FUNCTION time, so the name gets written to the catalog without the class being checked. Later when you SELECT using that function name, Flink tries to instantiate the class, gets ClassNotFoundException, falls back to PythonFunctionFactory, spawns python (not installed) → error=2.

Both errors have the same root cause: the JAR is not on the SQL Gateway session classpath. ADD JAR must be run with a path the Gateway container can see.

Register UDF tab — the 3-step wizard

The register tab is now a sequential wizard. The steps are ordered this way intentionally:

1
Step 1 — Verify classpath (do this after Upload JAR)
Click ⟳ SHOW JARS to see what JARs are currently on the session classpath. If the list is empty, the JAR was either not uploaded yet, or uploaded to a different container than the Gateway. Use the path input and ▶ ADD JAR button to manually load it. The expandable panel shows the three fix options for the separate-container case.
2
Step 2 — Function details
Fill in: Function Name, Language (Java/Scala/Python — never SQL), Scope (Temporary or Temporary System), and the full class path. The class path is case-sensitive: com.streamsstudio.udf.AlertSeverityEnricher. The SQL preview updates live below.
3
Step 3 — Preview & Register
Click ⚡ Register Function. Before executing CREATE FUNCTION, the wizard silently re-runs SHOW JARS. If the classpath is still empty, it tries ADD JAR automatically using the last uploaded path. If that also fails, it stops with exact fix instructions instead of letting Flink throw a confusing stack trace. On success it runs SHOW USER FUNCTIONS to verify registration.
💡
The Upload JAR tab comes before Step 1. The correct full sequence is: ⬆ Upload JAR (Studio runs ADD JAR automatically after upload) → then come to ＋ Register UDF → confirm Step 1 shows your JAR → fill Step 2 → click Register in Step 3. The Register tab wizard exists to catch the case where the automatic ADD JAR failed.

Tab Reference

⬆ Upload JAR tab — START HERE for JAR UDFs

This is the entry point for all JAR-based UDFs. Drag your shaded JAR or click to browse. Studio uploads it to the JobManager via POST /jars/upload, then immediately attempts ADD JAR in the current Gateway session. If ADD JAR succeeds, the JAR path is pre-filled in the Register tab's Step 1 input and the step 1 badge turns green. If ADD JAR fails (separate containers), the Upload tab shows exact docker cp and shared volume instructions.

📚 Library tab

Click ⟳ Refresh to run SHOW FUNCTIONS, SHOW USER FUNCTIONS, and SHOW VIEWS in parallel. Three sections: ◫ Session Views, ⨍ User UDFs, ⨍ Built-in Functions. Click any item to insert it at cursor.

⬡ Maven / Gradle tab

Generates a complete pom.xml or build.gradle with correct provided/compileOnly Flink dependency scopes. Fill in Group ID, Artifact ID, Flink version, Java version, and extra dependencies. Preview updates live.

UDF Types

f(x)

Scalar (UDF)

One row in → one value out. Classification, masking, format conversion. extends ScalarFunction.

⇶

Table (UDTF)

One row in → many rows out. Used with CROSS JOIN LATERAL TABLE(...). extends TableFunction.

∑

Aggregate (UDAGG)

Many rows in → one value out. Maintains an accumulator per group. Supports retraction. extends AggregateFunction.

◫

Pure SQL — no JAR, no classpath. Reusable named SELECT. Appears in SHOW VIEWS and the Library. Session-scoped.

Complete JAR UDF Workflow

1
Write the class — extend ScalarFunction, implement eval(). See ⊞ Templates for complete skeletons.
2
Generate build config — ⬡ Maven/Gradle tab → fill in Group ID, Artifact ID, Flink version → copy the generated pom.xml or build.gradle.
3
Build shaded JAR — mvn clean package -DskipTests → outputs target/artifact-shaded.jar. Must NOT bundle Flink core jars (use provided scope).
4
Upload JAR — ⬆ Upload JAR tab → drag the shaded JAR → click Upload. Studio runs ADD JAR automatically. Watch the result panel — if it shows green, the JAR is on the classpath and you can skip Step 1 in the wizard.
5
Confirm classpath — ＋ Register UDF tab → Step 1 → click ⟳ SHOW JARS. If your JAR is listed, proceed. If not, enter the path and click ADD JAR. See the expandable troubleshooting panel for the separate-container case.
6
Register — Step 2: fill function name (e.g. enrich_severity), class path (e.g. com.streamsstudio.udf.AlertSeverityEnricher), Language = Java. Step 3: click ⚡ Register Function.
7
Use in SQL — Library → ⟳ Refresh → click function name to auto-insert at cursor. SELECT enrich_severity(risk_score) FROM events;. ⚠ ADD JAR is session-scoped. You must re-run it every new session before calling the function.

Complete SQL View Workflow

1
Open View Builder — ◫ View Builder tab in UDF Manager. Select ◫ View mode.
2
Fill in the form — View Name (e.g. signals_enriched), Source Table (e.g. tower_telemetry), Computed Columns (e.g. quality AS CASE WHEN rssi > -70 THEN 'EXCELLENT' ELSE 'POOR' END), optional WHERE clause.
3
Create — click ⚡ Create View. The view is created in the current session. You can now query it: SELECT * FROM signals_enriched;
4
Verify — Library tab → ⟳ Refresh → your view appears in the ◫ Session Views section. Click it to insert a SELECT query into the editor.

🧪UDF Demo — AlertSeverityEnricher + Signal Quality View v1.2.0

A complete walkthrough demo using the telecom network monitoring dataset. Demonstrates both UDF paths: Path A registers AlertSeverityEnricher as a Java JAR UDF, and Path B creates a signals_enriched SQL view — both applied to a live datagen stream of tower telemetry. Copy each tab into Str:::lab Studio and run in order.

⨍
How to use this demo Create 7 tabs in Str:::lab Studio. Copy each block below into the corresponding tab. Run Tab 1 first, then Tab 2 (CREATE TABLE). For the JAR UDF path (Tabs 3–4), you need a compiled JAR — if you don't have one yet, skip to Tab 5 and use the pure SQL view path instead, which works with zero setup.

Session configuration. Run all statements together.

SET 'parallelism.default'                  = '2';
SET 'table.exec.mini-batch.enabled'        = 'true';
SET 'table.exec.mini-batch.allow-latency'  = '500 ms';
SET 'table.exec.mini-batch.size'           = '500';
SET 'table.exec.state.ttl'                 = '3600000';
SET 'table.exec.source.idle-timeout'       = '15000';

Register the tower telemetry source table. Run once per session.

-- ── Tab 2: Source table ─────────────────────────────────────────────────────

    'rows-per-second'                = '50',
    'fields.tower_id.min'            = '1',
    'fields.tower_id.max'            = '200',
    'fields.tower_raw_status.min'    = '0',
    'fields.tower_raw_status.max'    = '4',
    'fields.rssi_raw.min'            = '-120.0',
    'fields.rssi_raw.max'            = '-40.0',
    'fields.sinr_raw.min'            = '0.0',
    'fields.sinr_raw.max'            = '30.0',
    'fields.latency_ms.min'          = '5',
    'fields.latency_ms.max'          = '350',
    'fields.packet_loss_pct.min'     = '0.0',

SELECT tower_id, rssi_raw, sinr_raw, latency_ms, packet_loss_pct FROM tower_telemetry LIMIT 5;

PATH A continued. Use the registered JAR UDF in a query. Requires Tab 3 to have completed successfully.

-- ── Tab 4: Use JAR UDF in a query ───────────────────────────────────────────
-- Calls enrich_severity() to classify towers by signal degradation score.
-- The UDF takes a normalised 0.0–1.0 score (derived from packet_loss_pct).

SELECT
    event_id,
    tower_id,
    CASE tower_raw_status
        WHEN 0 THEN 'ACTIVE' WHEN 1 THEN 'DEGRADED'
        WHEN 2 THEN 'MAINTENANCE' WHEN 3 THEN 'CONGESTED'
        ELSE 'OFFLINE'
    END                                             AS tower_status,
    ROUND(rssi_raw, 1)                              AS rssi_dbm,
    ROUND(packet_loss_pct, 2)                       AS packet_loss_pct,
    -- Call the Java UDF: normalise packet_loss_pct to 0.0–1.0 range
    enrich_severity(packet_loss_pct / 15.0)         AS alert_severity,
    active_subscribers,
    event_ts
FROM tower_telemetry
LIMIT 20;

-- ── Expected output ──────────────────────────────────────────────────────────
-- alert_severity = CRITICAL when packet_loss_pct > 12.0  (>= 80% of 15)
-- alert_severity = HIGH     when packet_loss_pct >  8.25 (>= 55%)
-- alert_severity = MEDIUM   when packet_loss_pct >  4.5  (>= 30%)
-- alert_severity = LOW      otherwise

-- ── Troubleshooting ──────────────────────────────────────────────────────────
-- If you see: "Could not find function 'enrich_severity'"
--   → The function was not registered. Go back to Tab 3.
-- If you see ClassNotFoundException or Python error
--   → ADD JAR was not run. Go to Register UDF → Step 1 → ADD JAR.

PATH B — SQL View. No JAR required. Creates the same enrichment logic as a reusable temporary view. Works immediately after Tab 2.

◫
You can also create this view using the View Builder tab in UDF Manager Open ◫ View Builder, select ◫ View mode, set View Name = signals_enriched, Source Table = tower_telemetry, add the computed columns below, click ⚡ Create View. The SQL below is exactly what the View Builder generates. 
-- ── Tab 5: Create SQL View (no JAR needed) ──────────────────────────────────
-- This view encapsulates the same enrichment logic as AlertSeverityEnricher
-- but uses pure Flink SQL — no compilation, no upload, no classpath required.

CREATE TEMPORARY VIEW signals_enriched AS
SELECT
    event_id,
    tower_id,
    -- Decode numeric status to human-readable string
    CASE tower_raw_status
        WHEN 0 THEN 'ACTIVE'
        WHEN 1 THEN 'DEGRADED'
        WHEN 2 THEN 'MAINTENANCE'
        WHEN 3 THEN 'CONGESTED'
        ELSE 'OFFLINE'
    END                                             AS tower_status,
    ROUND(rssi_raw, 1)                              AS rssi_dbm,
    ROUND(sinr_raw, 1)                              AS sinr_db,
    latency_ms,
    ROUND(packet_loss_pct, 2)                       AS packet_loss_pct,
    active_subscribers,
    -- Signal quality index — same logic as AlertSeverityEnricher
    CASE
        WHEN rssi_raw > -70 AND sinr_raw > 20 AND packet_loss_pct < 1  THEN 'EXCELLENT'
        WHEN rssi_raw > -85 AND sinr_raw > 10 AND packet_loss_pct < 3  THEN 'GOOD'
        WHEN rssi_raw > -100 AND sinr_raw > 5 AND packet_loss_pct < 8  THEN 'FAIR'
        ELSE 'POOR'
    END                                             AS signal_quality,
    -- Alert severity — exactly matches AlertSeverityEnricher.eval() logic
    CASE
        WHEN packet_loss_pct / 15.0 >= 0.80 THEN 'CRITICAL'
        WHEN packet_loss_pct / 15.0 >= 0.55 THEN 'HIGH'
        WHEN packet_loss_pct / 15.0 >= 0.30 THEN 'MEDIUM'
        ELSE 'LOW'
    END                                             AS alert_severity,
    -- SLA classification
    CASE
        WHEN latency_ms < 20  AND packet_loss_pct < 0.5 THEN 'GOLD'
        WHEN latency_ms < 50  AND packet_loss_pct < 1.0 THEN 'SILVER'
        WHEN latency_ms < 100 AND packet_loss_pct < 3.0 THEN 'BRONZE'
        ELSE 'BREACHED'
    END                                             AS sla_class,
    event_ts
FROM tower_telemetry;

-- Verify: view is now registered in this session
SHOW VIEWS;

PATH B continued. Query the view — it behaves exactly like a table.

-- ── Tab 6: Query the SQL View ────────────────────────────────────────────────
-- The view is queried just like any table.
-- No function call syntax needed — enrichment is baked in.

-- Live stream: all enriched tower events
SELECT
    tower_id,
    tower_status,
    rssi_dbm,
    latency_ms,
    packet_loss_pct,
    signal_quality,
    alert_severity,
    sla_class,
    event_ts
FROM signals_enriched
LIMIT 30;

-- Filter: critical and high alerts only (equivalent to JAR UDF output in Tab 4)
SELECT
    tower_id,
    tower_status,
    packet_loss_pct,
    alert_severity,
    sla_class,
    event_ts
FROM signals_enriched
WHERE alert_severity IN ('CRITICAL', 'HIGH')
LIMIT 20;

-- Aggregation: alert count by severity (no window — batch-style count on LIMIT)
SELECT alert_severity, COUNT(*) AS cnt
FROM signals_enriched
GROUP BY alert_severity
LIMIT 100;

🎨Colour Describe New — v1.0.23

Colour Describe lets you visually highlight result rows in real time based on user-defined field conditions — as rows stream in from a live Flink query.

🎨
Two highlighting modes exist side-by-side ✦ Auto Colour applies built-in semantic rules based on column name patterns (status, risk_score, latency_ms…) — one click, no configuration. 🎨 Colour Describe lets you define your own rules for any column in any query. Both buttons are in the Results toolbar.

Activating Colour Describe

Click 🎨 Colour Describe in the Results toolbar. This opens a configuration modal. You must configure at least one rule and click ⚡ Apply & Activate before any highlighting occurs. When active, the button reads 🎨 Colour Describe ●.

Modal workflow

1

Select a query slot

A dropdown lists all active result slots. Select the slot to apply highlighting to. All column names auto-populate the Field dropdown.

2

Build a rule

Pick Field, Operator (== != > >= < <= contains starts ends regex), Value, and Label.

3

Pick a color

Eight semantic presets (red, amber, yellow, teal, blue, purple, green, orange) plus custom hex picker.

4

Choose a style mode

Row background, Left border accent, or Text color.

5

Add, reorder, delete rules

Rules evaluated top-to-bottom — first match wins per row. ↑↓ to reorder. No rule limit.

6

Apply & Activate

Click ⚡ Apply & Activate. Existing rows re-coloured immediately. New rows coloured as they stream in. Legend appears below the table with ✕ Turn off.
